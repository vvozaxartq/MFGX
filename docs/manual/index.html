    MFGX 系統簡介¶

    MFGX 是一套製造環境用的自動化測試系統（Automatic Test System），用於執行產品測試流程與品質控管。
    核心目標是：配方化流程（Recipe/TestPlan）+ 可擴展腳本（Scripts）+ 穩定的設備整合與資料記錄（Logging/Traceability）。

    1. 系統定位與適用場景¶

    MFGX 主要應用於產線/實驗室測試站台，支援： - 多設備整合：DUT、IO 卡、CCD 相機、Motion 馬達、通訊介面等 - 配方化流程：以 Recipe/TestPlan 組織整體測試步驟與節點 - 多 DUT 並行與圓盤（Rotary）型產線：包含多工位/輪轉/上下料節奏控制等

    2. 主要特點¶

    • 多設備支援（Multi-Device Support）：同時控制與測試多個 DUT（Device Under Test）。
    • 配方化流程（Recipe/TestPlan）：以樹狀流程定義節點與參數，降低「寫死流程」的維護成本。
    • 靈活腳本架構（Flexible Scripts）：以 Script 類別庫擴展功能（通訊、量測、判定、通用工具）。
    • 人性化操作介面 (User-friendly UI)：以直覺化的操作流程呈現測試步驟與提示資訊，支援單步/逐段執行以利工程除錯與快速定位問題，並可在流程中顯示影像/截圖（例如相機畫面、比對結果、錯誤示意），提升導入效率並降低現場誤操作風險。
    • 權限管理（Permission Management）：多層級使用者權限與操作限制。
    • 資料記錄（Data Logging / Traceability）：Log、CSV、影像、報表與備份路徑策略。

    3. 名詞與核心概念¶

    • Recipe / TestPlan：流程配方，定義容器與腳本節點、參數、Spec 與 FailJump/跳轉策略。
    • Script：最小功能單元，負責設備通訊、量測、判定或通用功能（例如檔案、字串、等待、比對等）。
    • Container：流程組織單元，負責「序列/並行/重試/條件/例外」等結構控制。
    • DUT：待測物（Device Under Test），通常對應一個裝置實例 + UI Dashboard + 測試狀態/資料容器。
    • Dashboard：每個 DUT 對應的 UI 顯示區（TreeView/DataGridView/Log/狀態燈等）。

    4. 系統架構（分層）¶

    MFGX 通常可視為「UI → 流程/配方 → 腳本引擎 → 設備整合 → 資料落地」的分層結構：

    MFGX 系統
├── UI Layer（使用者介面）
│   ├── 主視窗 / Login / 配方管理 / Dashboard
│
├── Business Layer（業務邏輯）
│   ├── RecipeManagement（配方載入/管理）
│   ├── Global / Context（全域狀態、環境、共用資源）
│   ├── DUT_BASE（DUT 生命週期/資料/狀態）
│   └── RotaryTestController（圓盤/多工位節奏協調）
│
├── Execution Layer（執行引擎）
│   ├── Container（流程結構：序列/並行/重試/條件）
│   └── ScriptBase（腳本節點：Pre / Process / Post）
│
├── Device Layer（設備整合）
│   ├── DUT Devices（被測裝置）
│   ├── IO Devices（DI/DO、Modbus 等）
│   ├── CCD Cameras（取像、判讀）
│   └── Motion（馬達、定位、教導）
│
└── Data Layer（資料層）
    ├── INI/JSON（配置/環境變數/配方）
    ├── Logs/CSV/Image（測試記錄/追溯）
    └── Reports/DB（報表/SQLite 等）

    5. 物件導向設計（OOP）與擴展點¶

    MFGX 的可維護性與可擴展性主要來自 OOP 的抽象/繼承/組合。

    5.1 核心抽象（抽象類別/介面）¶

    • DUT_BASE（抽象/基底類）
      定義 DUT 的共通行為：狀態、資料容器、UI Dashboard 綁定、測試生命週期（初始化/開始/結束/清理）。
    • ScriptBase（抽象/基底類）
      定義腳本節點的共通框架：參數取得、通用紀錄、Spec 檢查、錯誤處理、Pre/Process/Post 三段式。
    • Container（流程結構類）
      負責「如何跑」：序列、並行、重試、條件分支、FailJump/例外跳轉等流程控制。

    5.2 繼承（Inheritance）：用於「共通骨架 + 專案差異」¶

    典型做法： - 新增某產品 DUT：class DUT_XXX : DUT_BASE - 新增某功能腳本：class Script_YYY : ScriptBase

    建議：把共通流程與規範固定在基底類，子類只實作差異處（通訊、指令、量測與判定）。

    5.3 組合（Composition）：用於「設備/服務可替換」¶

    建議以組合方式讓 DUT/Script 取得服務： - DUT 持有（has-a）Service：CameraService、MotionService、IOService、CommService… - Script 透過 Context/ServiceLocator/DI 取得服務

    優點：設備更換或通訊方式替換時，變更集中在 Service，不會讓 DUT/Script 失控膨脹。

    5.4 典型關係¶

    Recipe/TestPlan
   └─ Container（控制流程：序列/並行/條件）
        └─ ScriptBase（節點骨架）
             └─ Script_xxx（專案/設備實作）

DUT_BASE（DUT骨架/資料/狀態）
   ├─ Dashboard（UI呈現）
   └─ Services/Devices（Camera/IO/Motion/Comm...）

    5.5 系統擴充方式¶

    • 新增 DUT：繼承 DUT_BASE，補上連線/初始化/資料與 UI 綁定策略
    • 新增 Script：繼承 ScriptBase，實作測項邏輯與 Spec 判定
    • 新增流程結構：擴展/配置 Container（並行、重試策略、FailJump 邏輯）
    • 新增設備整合：以 Service/Adapter 封裝通訊與操作（讓上層保持穩定）
    • 擴展點（Extension Point）：MFGX 的核心設計是「流程配方化 + 功能模組化 + 設備解耦」,可以「安全地新增功能」

    6. 系統需求¶

    6.1 軟體需求¶

    • Operating System：Windows 7 或更新版本
    • .NET Framework：.NET Framework 4.7.2（或更新版本，依專案實際為準）

    6.2 硬體需求（基本建議）¶

    • CPU：Intel Core i3 或更高
    • RAM：4GB 或更多（若含影像處理/多 DUT 建議更高）
    • Disk：至少 1GB（視 Log/影像量調整）
    • Display：1280×720 或更高
    • 外設：依站別需求（IO 卡、相機、馬達控制器、掃碼槍等）
